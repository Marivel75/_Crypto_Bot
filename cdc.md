# CDC - K3s mono-noeud + PostgreSQL (Helm) + Prometheus/Grafana

- Version: v0.1
- Date: 2026-02-15
- Cible: 1 VPS Ubuntu (mono-noeud) pour commencer, evolutif vers multi-noeuds plus tard

## 1) Objectif

Deployer une base **PostgreSQL** dans un environnement **Kubernetes maitrise** (K3s) sur un VPS Ubuntu, en **mono-noeud** au depart, avec une observabilite **Prometheus + Grafana**.

## 2) Perimetre

Inclus:
- Provisioning OS (SSH key only, root SSH off, firewall strict, user sudo)
- Installation K3s (runtime par defaut K3s, pas de "runtime engineering")
- Deploiement via Helm: `prometheus-community/kube-prometheus-stack` (Prometheus Operator + Prometheus + Grafana)
- Deploiement via Helm: PostgreSQL via un chart Helm (ex: `bitnami/postgresql`) en mono-instance
- Stockage persistant PostgreSQL via PVC (StorageClass par defaut K3s, typiquement `local-path`)
- Integration metrics PostgreSQL dans Prometheus (exporter + `ServiceMonitor` si Prometheus Operator)
- Automatisation Ansible idempotente (roles/playbooks) + validation post-deploiement

Hors perimetre:
- Changer/forcer le runtime (Docker/containerd) ou "rajouter containerd" comme chantier
- HA reelle PostgreSQL sur mono-noeud
- Stack logs (Loki/ELK/VictoriaLogs)
- GitOps complet (ArgoCD/Flux) - possible plus tard

## 3) Contraintes et principes

- Simplicite, reproductibilite, et evolution progressive (Day-0/Day-1/Day-2).
- CNI (v1): **Cilium** (Flannel desactive sur K3s).
- Ingress (v1): **Traefik** (packaged component K3s) pour HTTP/HTTPS.
- Reverse proxy PostgreSQL: aucun (PostgreSQL reste interne cluster; acces dev via SSH tunnel/port-forward).
- Exposition reseau minimaliste: par defaut WAN ouverts 22/80/443 uniquement (option homelab: +5432 restreint).
- API K8s 6443: non expose au WAN tant que possible (administration via tunnel SSH).
- PostgreSQL 5432:
  - mode par defaut: non expose au WAN (acces externe via SSH tunnel / port-forward)
  - mode homelab (autorise): expose sur le VPS en 5432 avec user/mdp (idealement TLS) et firewall restreint a ton IP
- PostgreSQL accessible en interne cluster (Service `ClusterIP`) par defaut.
- Secrets geres (idealement via Ansible Vault) et injectes au chart via `existingSecret` (quand supporte).

## 4) Architecture cible (v1 mono-noeud)

- Namespace `monitoring`: kube-prometheus-stack (Prometheus, Grafana, Alertmanager)
- Namespace `database`: PostgreSQL + Secrets + (option) ServiceMonitor
- Stockage: PVC RWO sur StorageClass par defaut K3s

## 5) Acces a PostgreSQL (depuis le cluster et depuis ton PC)

Dans le cluster (services deployee dans K3s):
- Les apps consomment PostgreSQL via un Service DNS interne (ClusterIP), credentials via Secret.

Depuis ton PC (dev) sans VPN (recommande: ne pas ouvrir 5432):

Option A (recommandee): SSH tunnel uniquement (port 22), sans exposer 5432 ni 6443 au WAN.
- Sur le VPS: faire un `kubectl port-forward` vers PostgreSQL en bind local (ex: `127.0.0.1:15432`).
- Sur ton PC (chez toi): faire un tunnel SSH `localhost:5432 -> vps:127.0.0.1:15432`.
- Ensuite: DBeaver et ton app Python se connectent a `127.0.0.1:5432` (le SSH transporte le trafic jusqu au VPS).

Exemple:
```bash
# VPS (le binaire kubectl est fourni via k3s)
sudo k3s kubectl -n database get svc
sudo k3s kubectl -n database port-forward svc/<svc_postgres> 15432:5432 --address 127.0.0.1

# PC
ssh -N -L 5432:127.0.0.1:15432 <user>@<vps_ip>
```

Pour DBeaver:
- Host: `127.0.0.1`
- Port: `5432`
- Database/user/password: ceux definis dans les Secrets Helm/PostgreSQL.

Pour une app Python locale:
- Connection string typique: `postgresql://<user>:<password>@127.0.0.1:5432/<db>`

Option B: `kubectl port-forward` depuis ton PC via un tunnel SSH vers l API K8s (6443) (toujours sans exposer 5432).

Exemple:
```bash
# PC
ssh -N -L 6443:127.0.0.1:6443 <user>@<vps_ip>
kubectl --kubeconfig <kubeconfig_k3s> -n database port-forward svc/<svc_postgres> 5432:5432
```

Option C (confort): rendre le port-forward "always-on" sur le VPS (systemd), et ne garder que le tunnel SSH cote PC.
- Avantage: DBeaver et ton app Python peuvent se connecter sans relancer `kubectl port-forward` a chaque fois.

Option D (homelab): exposer PostgreSQL directement sur `<vps_ip>:5432` (deconseille sur Internet, acceptable en homelab si restreint).
- Helm (Bitnami PostgreSQL): passer le service en `LoadBalancer` (K3s ServiceLB) pour binder 5432 sur le noeud.
- Firewall VPS: autoriser 5432/tcp uniquement depuis ton IP publique (ou un CIDR strict).
- Auth: utiliser un user applicatif dedie (eviter `postgres` a distance) + mot de passe long.
- TLS: recommande (auto-signe acceptable en homelab).

Exemple values (indicatif):
```yaml
auth:
  enablePostgresUser: false
  username: app
  password: "CHANGE_ME_LONG_RANDOM"
  database: appdb

tls:
  enabled: true
  autoGenerated: true

primary:
  service:
    type: LoadBalancer
    ports:
      postgresql: 5432
```

Connexion:
- DBeaver: Host `<vps_ip>`, Port `5432`, user/pass/db ci-dessus (SSL requis si TLS active).
- Python: `postgresql://app:...@<vps_ip>:5432/appdb?sslmode=require`

Ouverture 5432 (deconseille):
- Si vraiment necessaire temporairement: ouvrir 5432 uniquement a ton IP via firewall, imposer TLS, et limiter `pg_hba` aux CIDRs stricts.

## 6) Exigences Ansible (objets a prevoir)

Objets et patterns:
- Inventaire (mono-noeud aujourd hui, extensible vers groupes `servers`/`agents`)
- `group_vars/` + `host_vars/` (toutes les valeurs deploies)
- Roles:
  - `base_os` (hardening + packages)
  - `k3s` (install + config)
  - `monitoring` (kube-prometheus-stack)
  - `postgresql` (chart Helm + values)
  - `validation` (checks post-run)
- Templates Jinja2: `/etc/rancher/k3s/config.yaml`
- Templates Jinja2: `values.yaml` Helm (monitoring + postgres)
- Handlers (restart services si templates changes)
- Playbooks "phase": `provisioning`, `upgrade`, `reset`, `validation`

## 7) Criteres d'acceptation (DoD)

- OS:
  - SSH key only, root SSH off, firewall applique
- Cluster:
  - `kubectl get nodes` = Ready
  - CoreDNS OK
  - StorageClass par defaut OK + PVC test OK
- Monitoring:
  - Prometheus UP, targets scrape OK
  - Grafana UP
- PostgreSQL:
  - Pod Ready + PVC Bound
  - Service accessible en interne cluster
  - test read/write OK (depuis un Pod)
- Metrics PostgreSQL:
  - exporter actif
  - scrape OK (via ServiceMonitor si Prometheus Operator)
- Rejouabilite:
  - un 2e run Ansible ne casse rien (idempotence + validation)
